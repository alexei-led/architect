# architect

> Instruction-first architecture review plugin for AI coding runtimes. It ships a read-only Architect agent, architecture review/scorecard/plan skills, methodology guidance, report/plan templates, and Python helper CLIs for validating and comparing evidence-based architecture reports.

This file orients AI agents working in this repository. It follows the emerging `/llms.txt` convention: a concise Markdown guide with links to the highest-signal project files. This repo uses the requested singular filename `llm.txt`; for web or crawler discovery, also expose the same content as `llms.txt`.

## Repository map

- `src/agents/architect/AGENT.md` — the single source role prompt. No runtime frontmatter in this source file.
- `src/skills/*/SKILL.md` — activation-scoped procedures for review, planning, scoring, methodology, and tool usage.
- `src/templates/` — report, plan, and scorecard contracts used by the agent and validators.
- `src/architect_tools/` — Python CLIs: `architect-doctor`, `architect-validate-report`, and `architect-compare-reports`.
- `src/plugins/architecture/plugin.yaml` — package manifest declaring what ships.
- `docs/` — explanatory docs mirroring the source contracts. Source files win if docs disagree.
- `tests/` — contract, scaffold, skill-eval, template, and CLI tests.
- `scripts/git-hooks/` — local pre-commit/pre-push hooks wired by `make setup`.

## Agent operating rules

- Treat production source code as read-only when acting as the Architect role. The Architect reviews, reports, and plans; it does not rewrite systems.
- Write only approved architecture reports and refactoring plans, using the templates verbatim as skeletons.
- Separate facts, hypotheses, scores, and recommendations. Every finding needs an evidence reference.
- Establish intended architecture before judging observed architecture. Source order: user interview, docs/ADRs/README, manifests/config, directory structure, inferred clusters.
- Never score from directory shape alone. Use observed behavior, dependency evidence, graph/search output, commands, and cited files.
- Apply `src/templates/scorecard.yaml` before assigning scores. Do not invent dimensions, bands, confidence labels, or comparability keys.
- Missing or failed tools are coverage gaps, not fatal errors. Record the confidence impact instead of pretending coverage exists.
- Recommend incremental refactoring only: seams, characterization tests, boundary repair, and fitness checks. No big-bang rewrites.

## Primary workflows

### Architecture review

1. Interview for missing intended architecture and constraints.
2. Build a system map before quality judgment.
3. Gather cited evidence across discovery, structural, semantic, dependency, change, operational, security, and report dimensions.
4. Apply the scorecard skill and rubric contract.
5. Produce a report from `src/templates/report.md`.
6. Validate with `architect-validate-report`.

### Refactoring plan

1. Start from report findings and evidence refs.
2. Keep one hotspot, boundary, or flow per plan unless a roadmap is requested.
3. Use `src/templates/plan.md`.
4. Include success criteria, phases, preconditions, postconditions, verification checks, and safety notes.
5. Pair each boundary repair with a fitness check so the boundary cannot silently rot again.

### Report comparison

- Compare only reports with matching `scope`, `rubric_version`, and `tool_coverage_level`.
- If comparability keys differ, emit the non-comparability reason. Do not invent a trend.

## Local commands

```sh
make setup                         # install hooks and dev dependencies with uv
make lint                          # ruff check + ruff format --check
make test                          # pytest
make check                         # lint + test
uv run architect-doctor --repo .    # inspect local analysis tool availability
uv run architect-validate-report path/to/report.md
uv run architect-compare-reports base-report.md head-report.md
```

## Development conventions

- Python 3.12+ project managed by `uv`.
- Keep instruction files concise, activation-scoped, and testable.
- Skill files require YAML frontmatter with `name` and `description`, and a `## When to use` section.
- The source Architect role prompt must start with `# Architect` and must not have runtime frontmatter.
- Template paths referenced by prompts/skills must resolve under `src/templates/`.
- Avoid placeholder markers in agent and skill instruction files; tests reject them.
- Comments should be short and useful. Dead code should be removed, not parked.

## Methodology constraints

- Balanced Coupling evaluates integration strength, distance, and volatility. Coupling is not automatically bad; imbalance is the problem.
- Architecture fitness means executable checks that can fail the build. Documentation alone does not raise the fitness score.
- Balanced Coupling material is summary-only with attribution. Do not paste or closely paraphrase external reference text into this repo.

## Verification checklist for agents

- Run `make check` before claiming implementation work is done.
- For report artifacts, run `uv run architect-validate-report <report>`.
- For changes to skills, role prompts, templates, or docs, inspect tests under `tests/` for contract expectations before editing.
